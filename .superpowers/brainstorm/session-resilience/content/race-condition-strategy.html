Resiliência: Race Conditions & Falhas de Persistência

Analisando os adapters de LocalStorage, identifiquei uma vulnerabilidade crítica de Race Condition.

A Vulnerabilidade Atual

O método salvar() segue o padrão: Ler → Modificar → Escrever.

// ABA 1 (Usuário A)
const todas = await listarTodas() // Retorna [T1]

// ABA 2 (Usuário B) - No mesmo milissegundo
const todas = await listarTodas() // Também retorna [T1]

// ABA 1 salva T2
localStorage.setItem('...', [T1, T2])

// ABA 2 salva T3 (Sobrescrevendo a T2!)
localStorage.setItem('...', [T1, T3]) 💥

Resultado: Perda silenciosa de dados se o app for usado em múltiplas abas ou muito rápido.

Estratégias de Mitigação

Qual nível de sofisticação você deseja aplicar para resolver este problema estrutural?

A

Sincronização Passiva (Storage Events)

Ouvir o evento window.addEventListener('storage') para reagir quando outra aba modificar o LocalStorage. Simples, mas não impede a sobrescrita, apenas atualiza a tela.

B

Bloqueio Otimista (Optimistic Locking)

Adicionar um número de versao ou timestamp ao pacote de dados. Se a versão que a Aba 1 tentar salvar for menor que a versão atual no banco, a gravação é rejeitada (Retry).

C

Travamento Mútuo (Web Locks API / Mutex)

Usar a API nativa de Web Locks (navigator.locks.request) para garantir que o bloco "Ler-Modificar-Escrever" seja atômico em todas as abas. É a solução mais segura e moderna.
